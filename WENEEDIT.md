# Дополнительная информация для 1 пукнта: 
## Когда лучше писать снапшот в БД
Кратко по стратегиям (их часто комбинируют):
1. Debounced autosave по изменению состояния
    Один источник правды: при любом изменении elements (или сериализованного документа) ставите таймер на 300–1000 ms и один запрос на «пачку» правок. Двойной клик даст одно логическое изменение выбора/настроек и один отложенный запрос (если не сбрасывать дебаунс слишком агрессивно — обычно как раз сброс таймера на каждое изменение даёт один flush после паузы).
2. Сохранять только при «коммите» жеста
    Для drag: вызывать API только если был заметный сдвиг (порог в px в координатах экрана/SVG), иначе считать это кликом/двойным кликом. Это правка в DraggablePath: передавать в onDragEnd флаг «был ли move» или дельту.
3. Явные точки сохранения
    После завершения DIY-пути (замыкание контура в handleSvgClick), после «Add Path», после закрытия панели настроек с «Применить» — плюс опционально дебаунс для всего остального.
4. Не дублировать с drag
    Если введёте дебаунс на уровне store, onDragEnd может только сбрасывать локальный флаг или вызывать flushPendingSave(), а не полный updateProject каждый раз.
5. Перед уходом со страницы
    beforeunload / visibilitychange — сбросить дебаунс и отправить последнюю версию (осторожно с sendBeacon или синхронным XHR для надёжности при закрытии вкладки).
## Практическая рекомендация для вашего редактора
Основной поток: подписка на изменение документа (elements + размеры) → debounce 500–800 ms → один updateProject. Так вы не зависите от того, pointer это или клик, и двойной клик не даст два мгновенных запроса подряд.
Дополнительно: в DraggablePath не вызывать «тяжёлый» колбэк на каждый pointerup, а только если был pointermove с дельтой > ε, либо убрать дублирование pointerleave / разделить логику «отмена drag при leave» и «commit drag».
