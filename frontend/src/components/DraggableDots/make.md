# Что найдено по багу с symmetric у замыкающего узла

## Кратко

Тип узла, скорее всего, **не сбрасывается** перед перемещением. Проблема выглядит как конфликт логики в `handlePointerMove` внутри `DraggableDots`.

## Подробно

При перетаскивании направляющей (`guideLine`) выполняется ветка:

- `if (targetDotIndex === dotsArr.length - 2 || targetDotIndex === 0) { ... }`

Внутри неё есть специальная обработка для замыкания контура (работа с `dotsArr[0].out` для замыкающего узла в `symmetric`).

Но сразу после этой ветки идёт **вторая, общая обработка**:

- `if (initialPosRef.current.type === 'symmetric') { ... } else if (...) { ... }`

Она выполняется **в любом случае**, в том числе для того же замыкающего узла, и работает уже с `obj.out`, а не с `dotsArr[0].out`.

## Почему это ломает ожидаемое поведение

Для замыкающего узла (в твоём примере 4-й узел) противоположная ручка фактически живёт на нулевом узле (`dotsArr[0].out`).  
Специальная ветка это учитывает, но общая ветка тут же применяет обычную логику и фактически “перетирает” результат/ожидаемую связку.

Из-за этого создаётся ощущение, что тип `symmetric` сбросился, хотя корень проблемы — в двойном применении разных правил.

## Дополнительное наблюдение

В `handlePointerDown` для замыкающего узла берётся:

- `type` из `dotsArr[objIndex].type`
- `out` из `dotsArr[0].out`

Это выглядит осознанно и похоже на корректную идею для замкнутого path.  
То есть основная подозрительная зона не тут, а именно в повторной обработке в `handlePointerMove`.

## Что изменено

В `DraggableDots` внесены правки в `handlePointerMove`:

1. Общая ветка обработки направляющих теперь не отрабатывает повторно для спец-кейса замыкающего узла (чтобы не перетирать результат).
2. Спец-ветка для связки `последний узел <-> узел 0` дополнительно ограничена только замкнутым контуром:
   - добавлена проверка `isClosedShape`
   - добавлен флаг `isSharedClosingNode`
3. За счёт этого логика `symmetric` для замыкающего узла применяется один раз и корректно двигает связанную ручку у узла `0`.

## Дополнительный фикс после повторной проверки

После повторной диагностики найден ещё один источник проблемы:

- `DraggableDots` строил `dotsArr` из `parsePathData(customEll.d)`, из-за чего типы узлов могли пересчитываться и терять исходную семантику для shared-узла `0/last`.

Что дополнительно изменено:

1. Источник точек переключен на `customEll.points` (если есть), с fallback на `parsePathData(customEll.d)`.
2. В `handlePointerDown` для замкнутого контура и индексов `0`/`last` (`dotsArr.length - 2`) начальное состояние теперь всегда берётся из общей пары:
   - `in` от последнего узла
   - `out` от нулевого узла
   - `type` от последнего узла
3. В special-ветке `handlePointerMove` для shared-узла обновление теперь явно идёт через:
   - `sharedInNode = dotsArr[last]`
   - `sharedOutNode = dotsArr[0]`
   Это убирает неоднозначность, когда для `targetDotIndex === 0` мог обновляться не тот `in`-handle.
